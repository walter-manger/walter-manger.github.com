Docker Basics

I’ve been using Docker for about two years now, but I have only ever been a consumer of other’s configurations. On our team, we have someone that is dedicated to Docker configuration and orchestration – the developers are consumers of this through a collection of bash scripts. I have been uncomfortable with this since the beginning, but there has not been time for personal education in this area. Most of the working day for developers is dedicated to shipping features and bug fixes, with the rare occurance of learning how to better implement a feature. This is probably the case for most teams.

I recently got a free subscription to Linux Academy and saw that they were offering a course called Docker Quick Start, so I tried it out. This is what I’ve learned so far.

What is it?

Docker is a software container platform. In few words, Docker allows software to run in configured, isolated containers, to foster reproducable results.

But, why?

It is the cure for It works on my machine and it also simplifies automated deployments. Especially since there are cloud providers (AWS, GCP) that offer container services.

Concepts

You run a docker image, an instance of that image becomes a container.

Images

A Docker image is set of OS layers.

Containers

A Docker container is an instance of a Docker image.

Survive With These Commands

docker images

Show a list of available images to create containers from.

docker ps

Show a list of running containers. Use docker ps -a to show all containers (running or not).

docker exec

docker attach {containerID}

Attach to a docker container to run commands inside of the container. (Be careful, exit will stop the container from running. In most cases, using docker exec -it would be a better option.

docker rmi

docker rm docker ps -a -q

docker inspect {containerID | name}

docker run -d image:tag

-p vs -P -d vs -D -v (volume mounting)

docker build -t image . // Dockerfile location .

docker stop image
